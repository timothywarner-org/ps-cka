2. kind-down.ps1 — Shutdown Script Flow

Teaching point: Teardown order matters. The script deletes the cluster before stopping Docker, and stops Docker before shutting down WSL2. Reversing this order would cause errors.

3. KIND Cluster Architecture — What Gets Created

Teaching point: KIND runs Kubernetes nodes as Docker containers, not VMs. Each container runs systemd + kubelet + containerd (nested). The host's Docker daemon is the "hypervisor" equivalent. This is why KIND is fast but not production-grade.

7. Side-by-Side: KIND vs kubeadm on VMs

Teaching point: KIND is for fast iteration and daily practice. kubeadm on VMs is for exam-realistic experience. Use both: KIND 80% of the time, VMs when practicing cluster setup/teardown.

KIND (Containers)

  • Startup: ~2-4 minutes
  • RAM: ~2-3 GB total
  • Nodes: Docker containers
  • Runtime: containerd (nested in Docker)
  • CNI: kindnet (simple, built-in)
  • kubeadm: Runs automatically inside KIND
  • Networking: Docker bridge, port mappings
  • Best for: Daily practice, pod/service exercises, RBAC, storage
  • Limitation: No real VM isolation, can't practice cluster bootstrap

kubeadm on VMs

  • Startup: ~10-15 minutes
  • RAM: ~6-8 GB total (2GB per VM)
  • Nodes: Full Hyper-V VMs
  • Runtime: containerd (native)
  • CNI: Calico or Flannel (your choice)
  • kubeadm: You run init/join manually
  • Networking: Real private network (192.168.56.0/24)
  • Best for: Cluster setup practice, networking, exam simulation
  • Limitation: Heavier resources, slower iteration

What KIND Does Under the Hood (It's kubeadm!)

sequenceDiagram
    participant User
    participant KIND as KIND CLI
    participant Docker
    participant Container as Node Container
    participant kubeadm

    User->>KIND: kind create cluster --config yaml
    KIND->>Docker: Pull kindest/node image
    KIND->>Docker: Create container per node role
    Docker-->>Container: Start container with systemd
    KIND->>Container: Write kubeadm config patches
    Container->>kubeadm: kubeadm init (control-plane)
    kubeadm-->>Container: API server, etcd, scheduler running
    KIND->>Container: Install kindnet CNI
    loop Each worker
        Container->>kubeadm: kubeadm join
    end
    KIND->>User: Write kubeconfig, set context
    User->>User: kubectl get nodes (Ready!)
Key insight: KIND uses kubeadm internally! The difference is that KIND automates and containerizes it, while the VM path gives you hands-on kubeadm experience. Both produce a real Kubernetes cluster with the same API.
