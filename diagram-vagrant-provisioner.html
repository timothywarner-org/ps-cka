How Vagrant works (the mental model)

Vagrant is a declarative wrapper around a hypervisor. You write a single file (Vagrantfile) that describes what VMs you want and what should be on them. Vagrant translates that into Hyper-V API calls, runs shell scripts inside each VM, and tracks which VMs belong to which project in a hidden .vagrant/ directory. Three things that confuse everyone at first:
  1. The Vagrantfile is the source of truth. Vagrant re-reads it on every command. Edit the Vagrantfile, and the next vagrant up sees the new config — but provisioners don't re-run automatically on already-created VMs unless you explicitly say vagrant provision.
  2. .vagrant/ is Vagrant's brain. Inside it, machines/<vm>/<provider>/id holds the Hyper-V VM ID — the link between "the line that says control1 in your Vagrantfile" and "the actual VM running on Hyper-V." Delete .vagrant/ and Vagrant forgets your VMs exist (the VMs themselves keep running; Vagrant just can't talk to them).
  3. Provisioners run once at create, not on every boot. vagrant up on an existing VM is a fast boot. vagrant up --provision forces them to re-run. vagrant up --no-provision is explicit "don't even try" (used by cka-up.ps1 for speed).

That's the gotcha that bit us earlier: When the repo moved from L:\cka-lab to C:\github\ps-cka\src\cka-lab, the .vagrant/ directory came with it — but the VHDX files in D:\VMS\ didn't move, and Hyper-V kept seeing the VMs at their original IDs. Result: Vagrant prints a "this machine used to live in L:\cka-lab" warning, but the VMs are still findable because Hyper-V is the source of truth for actually-running VMs. Vagrant tracks; Hyper-V owns.

Inspect the provisioner state from inside a VM

# Inside the VM (vagrant ssh control1)

# Step-by-step record of what the provisioner did, with timestamps
sudo cat /var/log/cka-provision.log

# Just the >>> step headers (skim version)
sudo grep '>>>' /var/log/cka-provision.log

# Re-run a single provisioner stage (from the Windows host)
#   (idempotent — re-running on a clean VM is a no-op)
vagrant provision control1

Why this design

  • Provisioners are idempotent. Every step detects "already done" and no-ops on re-run. vagrant provision <vm> can be run any time without breaking state.
  • Each step is logged. /var/log/cka-provision.log on each VM is a step-by-step record with timestamps. If SSH drops mid-provision (a known Hyper-V + br_netfilter issue), you can SSH back in and read the log to see how far the script got on the guest.
  • The two red steps are the high-stakes ones. Step 5 (SystemdCgroup = true) is the single most common cause of kubeadm init failures in the wild. Step 8 (apt-mark hold) is what prevents a future apt upgrade from silently breaking the cluster.
